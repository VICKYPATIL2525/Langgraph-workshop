This is an educational project designed for a workshop to teach participants how
to use LangGraph - a framework for building structured AI workflows using Large
Language Models (LLMs). The project demonstrates key concepts through progressive
examples, from simple to complex workflows.

- Multiple tutorial examples (test0.py through test5.py)
- Both simple and commented versions for learning
- Real-world use cases including multi-database integration
- MongoDB and SQLite database integration examples

EXAMPLE 0: BASIC ADDITION WORKFLOW (Beginner)
──────────────────────────────────────────────────────────────────────────────
Files: test0.py, test0_commented_v1.py

Difficulty: ⭐ (Beginner)
Purpose: Introduce fundamental LangGraph concepts

Key Concepts Demonstrated:
  1. StateGraph: Creating a workflow graph
  2. TypedDict: Defining state structure
  3. Node functions: Processing units in the workflow
  4. Edges: Connecting nodes in sequence
  5. START/END: Special marker nodes
  6. workflow.compile(): Converting graph to executable
  7. workflow.invoke(): Running the workflow

Workflow Structure:
  START → [add node] → END

How It Works:
  - Takes two numbers (num1, num2) as input
  - The 'add' node performs addition: result = num1 + num2
  - Returns the result

State Structure:
  {
    'num1': float,      # First number
    'num2': float,      # Second number
    'result': float     # Sum of both numbers
  }

Example Execution:
  Input: {'num1': 5, 'num2': 5}
  Output: {'num1': 5, 'num2': 5, 'result': 10}

Key Learning Points:
  - Simplest possible LangGraph workflow
  - Shows state flow through nodes
  - Demonstrates workflow execution cycle

EXAMPLE 3: PARALLEL LLM ANALYSIS (Intermediate → Advanced)
──────────────────────────────────────────────────────────────────────────────
Files: test3.py, test3_commented_v1.py

Difficulty: ⭐⭐⭐⭐ (Advanced)
Purpose: Demonstrate true parallel execution of multiple nodes

Key Concepts Demonstrated:
  1. Parallel Nodes: Multiple nodes executing simultaneously
  2. Node Specialization: Each node focuses on one task
  3. Partial State Updates: Nodes return only their updates
  4. State Merging: LangGraph merges multiple updates automatically
  5. Synchronization: Waiting for all parallel nodes to complete
  6. Result Aggregation: Combining results from multiple sources

Workflow Structure:
        [START]
       /   |   \
      /    |    \
     ↓     ↓     ↓
  [Grammar] [Sentiment] [Clarity]   ← All run in parallel!
     \     |     /
      \    |    /
       \   |   /
        ↓ ↓ ↓
     [Finalizer]
        ↓
       [END]

How It Works:
  Three parallel analysis nodes run simultaneously on an essay:
    1. Grammar Node: Analyzes grammar, returns score
    2. Sentiment Node: Analyzes emotional tone, returns score
    3. Clarity Node: Analyzes understandability, returns score

  Finalizer Node (runs after all parallel nodes complete):
    - Receives all three scores
    - Generates comprehensive analysis report
    - Combines individual scores into final assessment

State Structure:
  {
    'essay': str,               # Essay to analyze (input)
    'grammar_score': int,       # 0-100 score (updated by grammar_node)
    'sentiment_score': int,     # 0-100 score (updated by sentiment_node)
    'clarity_score': int,       # 0-100 score (updated by clarity_node)
    'final_result': str         # Report (updated by finalizer)
  }

Example Execution:
  Input Essay: "Artificial intelligence is revolutionizing education..."

  Parallel Analysis Results:
    Grammar Score: 92/100
    Sentiment Score: 85/100
    Clarity Score: 88/100
    Average: 88.3/100

  Final Report: "Comprehensive analysis with strengths and improvements..."

Key Learning Points:
  - True parallel execution in LangGraph
  - Efficiency gains from parallel processing
  - State merging for concurrent operations
  - Building scalable workflows
  - Combining results from multiple analyses

Important Pattern:
  - Each parallel node returns ONLY what it changes: {'field': value}
  - LangGraph automatically merges all partial updates
  - Prevents conflicts between concurrent nodes

──────────────────────────────────────────────────────────────────────────────
File: test4_commented_v1.py

Difficulty: ⭐⭐⭐ (Intermediate)
Purpose: Demonstrate conditional branching in workflows

Key Concepts Demonstrated:
  1. Conditional Edges: Routes based on state values
  2. Decision Functions: Custom logic to determine next node
  3. Branching Workflows: Different paths for different inputs
  4. Dynamic Routing: Workflow path determined at runtime

Workflow Structure:
  START → [check_feedback] →
              ├─(positive)→ [thank_you] → END
              └─(negative)→ [apology] → END

How It Works:
  Step 1 - Sentiment Analysis:
    - Takes customer feedback as input
    - Uses LLM to determine sentiment (positive/negative)
    - Updates state with sentiment

  Step 2 - Conditional Routing:
    - If sentiment == "positive" → Thank the customer
    - If sentiment == "negative" → Apologize and help

  Step 3 - Response Generation:
    - Thank You (for positive): Generates thank you message
    - Apology (for negative): Generates apology and offers help

State Structure:
  {
    'feedback': str,    # Customer's original feedback
    'sentiment': str,   # "positive" or "negative"
    'response': str     # Generated response message
  }

Example Execution 1 (Positive):
  Input: {'feedback': 'I love this product! Very good.'}
  Path: check_feedback → thank_you → END
  Output: {'feedback': '...', 'sentiment': 'positive',
           'response': 'Thank you for your positive feedback...'}

Example Execution 2 (Negative):
  Input: {'feedback': 'This product is terrible. Worst ever.'}
  Path: check_feedback → apology → END
  Output: {'feedback': '...', 'sentiment': 'negative',
           'response': 'We sincerely apologize for this experience...'}

Key Learning Points:
  - Dynamic workflow routing
  - Conditional logic in graphs
  - Handling multiple workflow branches
  - Decision function patterns
  - Response customization based on context

EXAMPLE 5: HUMAN-IN-THE-LOOP ITERATIVE WORKFLOW (Advanced)
──────────────────────────────────────────────────────────────────────────────
Files: test5.py, test5_commented_v1.py

Difficulty: ⭐⭐⭐⭐ (Advanced)
Purpose: Demonstrate human approval loops and iterative refinement

Key Concepts Demonstrated:
  1. Human-in-the-Loop: Human reviews and approves AI output
  2. Iterative Refinement: Looping until approval
  3. Feedback Integration: Using human feedback to improve output
  4. Attempt Tracking: Monitoring iteration count
  5. Loop Breaking: Exiting based on conditions
  6. Interactive Workflows: Prompting for human input

Workflow Structure:
  START → [generate] → [approval] →
              ↑           ↓ (if approved → END)
              └───────────┘ (if not approved → loop back)

How It Works:
  Step 1 - Generate Description:
    - If first attempt: Generate initial product description
    - If rejected: Generate improved description using feedback
    - Update attempt counter

  Step 2 - Get Approval:
    - Display current description to human
    - Ask for approval (yes/no)
    - If approved: Set approved=True → EXIT
    - If rejected: Get feedback → LOOP BACK to generate

  Step 3 - Conditional Loop:
    - If approved: Go to END (workflow complete)
    - If not approved: Go back to generate with feedback

State Structure:
  {
    'product_name': str,    # Product to describe
    'description': str,     # Current generated description
    'approved': bool,       # Human approval status
    'attempts': int,        # Number of generation attempts
    'feedback': str         # Human's improvement feedback
  }

Example Execution (Multiple Iterations):
  Iteration 1:
    Generated: "Smart Watch - A wearable device"
    Human: "No, make it more appealing"
    Feedback: "more appealing"
    Loop back...

  Iteration 2:
    Generated: "Smart Watch - Advanced wearable with health tracking"
    Human: "Yes, approved!"
    Approved: True → END

Key Learning Points:
  - Building interactive AI systems
  - Handling user feedback in workflows
  - Iterative improvement patterns
  - Loop control in graphs
  - User engagement with AI systems
  - Practical applications requiring human oversight

Safety Features:
  - Auto-approval after 3 attempts (prevents infinite loops)
  - Attempt counting for monitoring
  - Clear user prompts and feedback collection

File: multisource_langgraph.py

Difficulty: ⭐⭐⭐⭐⭐ (Advanced)
Purpose: Real-world example integrating multiple databases with LLM routing

Key Features:
  1. Intent-Driven Execution: LLM determines which database to query
  2. Multi-Database Support: Both MongoDB and SQLite
  3. Query Routing: Automatic routing to correct database
  4. JSON-based Intent: Structured intent specification
  5. Error Handling: Graceful handling of invalid requests
  6. Data Integration: Combining data from different sources

Workflow Structure:
  START → [planner] → [executor] → [finalizer] → END

How It Works:
  Step 1 - Planner (Intent Recognition):
    - Takes user natural language question
    - Uses LLM to understand intent
    - Returns structured JSON with source and action

  Step 2 - Executor (Data Retrieval):
    - Routes to MongoDB or SQLite based on intent
    - Executes appropriate query
    - Returns results

  Step 3 - Finalizer (Response Generation):
    - Uses LLM to format results into natural language
    - Returns human-readable answer

Supported Intents:

  MongoDB Queries:
    1. Get Hostel Room:
       Intent: {"source": "mongo", "action": "get_hostel",
                "student_name": "student name"}
       Example Q: "What is the hostel room for Rohit Kumar?"

    2. Pending Fees:
       Intent: {"source": "mongo", "action": "pending_fees"}
       Example Q: "Which students have pending fees?"

  SQLite Queries:
    1. Top Scorers:
       Intent: {"source": "sqlite", "action": "top_scorers", "limit": 3}
       Example Q: "List the top 3 scorers"

    2. Timetable:
       Intent: {"source": "sqlite", "action": "timetable"}
       Example Q: "Show the timetable for MCA"

State Structure:
  {
    'question': str,        # User's natural language question
    'intent': dict,         # Parsed intent from LLM
    'db_result': Any,       # Query results from database
    'final_answer': str     # Formatted answer for user
  }

Example Execution:
  User Q: "What is the hostel room for Rohit Kumar?"

  Planner Output:
  {"source": "mongo", "action": "get_hostel", "student_name": "Rohit Kumar"}

  Executor Query: MongoDB find_one with regex on name
  Result: {"name": "Rohit Kumar", "hostel_room": "A-101"}

  Finalizer Output: "Rohit Kumar is assigned to hostel room A-101"

Key Learning Points:
  - LLM-driven workflow routing
  - Multi-database integration
  - Intent parsing and execution
  - Practical database query systems
  - Error handling and validation
  - Building intelligent query systems

Database Setup:
  - MongoDB: langgraph_db database with students, faculty, attendance collections
  - SQLite: institute.db with courses, subjects, exam_results, timetable tables

================================================================================
                        KEY CONCEPTS & TERMINOLOGY
================================================================================

State:
  - A TypedDict that defines data flowing through the graph
  - Persists across all nodes in a workflow
  - Each node receives and can modify the state
  - Flows forward through the graph execution

Node:
  - A function that processes the state
  - Receives current state as input
  - Performs some computation or action
  - Returns updated state (or partial state updates)
  - Can access LLMs, databases, or external services

Edge:
  - Connection between two nodes
  - Defines the flow of execution
  - Can be direct (simple edge) or conditional (based on state)

Graph:
  - Collection of nodes connected by edges
  - Represents the complete workflow
  - StateGraph is the main class for building graphs

Workflow:
  - The compiled, executable version of a graph
  - Created by calling graph.compile()
  - Executed with workflow.invoke(initial_state)

START/END:
  - Special built-in nodes marking graph entry/exit points
  - START is always the entry point
  - END is the normal exit point

Parallel Execution:
  - Multiple nodes starting from the same predecessor
  - All execute simultaneously (conceptually)
  - All must complete before dependent nodes run
  - Useful for independent tasks on same data

Conditional Routing:
  - Branching based on state values
  - Uses conditional_edges with decision functions
  - Different paths for different inputs
  - Dynamic workflow execution at runtime

Human-in-the-Loop:
  - Workflow pauses for human decision/approval
  - Human provides feedback or approval
  - Workflow continues based on human input
  - Useful for quality assurance and refinement loops

Intent:
  - High-level understanding of what action to take
  - Often generated by LLM from natural language input
  - Structured representation of user intent
  - Used for routing to appropriate handlers
